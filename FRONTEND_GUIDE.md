# FRONTEND_GUIDE

Гайд для фронтенд-команды проекта «Цифровой экзамен».
Этот документ — общий контекст для всех подсистем. Конкретные экраны
описаны в отдельных ТЗ (см. папку docs/). Здесь — как устроен проект и
как в нём работать.

> Если работаешь с ИИ-ассистентом — дай ему этот гайд + своё ТЗ. Тогда он
> будет подсказывать по нашей структуре, а не выдумывать свою.

---

## 1. О проекте

Веб-приложение для проведения семестровых работ в школе: рассадка,
проверка, аналитика. Роли: ученик, учитель, оператор, администратор —
у каждой свой кабинет.

**Стек:**
- React + TypeScript
- Vite — сборка и dev-сервер
- Tailwind CSS + shadcn/ui — стили и компоненты
- React Router — маршрутизация

Понадобятся (ставить по мере необходимости, согласовав с лидом):
- TanStack Query — запросы к API
- react-hook-form + zod — формы и валидация
- react-katex — рендер LaTeX (банк задач)
- react-pdf — просмотр PDF (сканы работ)

---

## 2. Структура папок

```
src/
  features/                  ← подсистемы (каждый фронт владеет своими)
    landing/                 публичная страница
    auth/                    вход
    settings/                настройки аккаунта
    contingent/              Контингент (Фронт 3)
    seating/                 Рассадка
    works/                   Работы (Фронт 1)
    bank/                    Банк задач (Фронт 2)
    processing/              Обработка ЭМ (Фронт 2)
    classrooms/              Аудитории (Фронт 3)
  components/
    ui/                      компоненты shadcn — НЕ редактировать вручную
    layout/                  шапка и общий каркас (AppHeader, AppLayout)
    shared/                  общие компоненты (PageHeader, состояния и т.д.)
  lib/                       утилиты, общий мок авторизации (mockAuth)
  App.tsx                    корневой роутер
```

**Правило:** каждый работает в своей папке `features/<подсистема>/`.
Внутри держи: страницы (`*.tsx`), запросы (`api.ts`), типы (`types.ts`),
моки (`mocks.ts`). Общее (нужное нескольким) — в `components/shared/`.


## 4. Как добавить новый экран

На примере экрана «Список работ ученика» (маршрут /student/works):

1. Создай файл в своей фиче:
   `src/features/works/StudentWorksPage.tsx`
   ```tsx
   export function StudentWorksPage() {
     return <div>...</div>
   }
   ```

2. Подключи маршрут в `src/App.tsx`:
   - добавь импорт:
     `import { StudentWorksPage } from "@/features/works/StudentWorksPage"`
   - найди заглушку этого маршрута и замени её:
     ```tsx
     // было:
     <Route path="/student/works" element={<PlaceholderPage title="Мои работы" />} />
     // стало:
     <Route path="/student/works" element={<StudentWorksPage />} />
     ```

3. Открой страницу в браузере (переключив роль, см. п.9).

Алиас `@/` означает папку `src/`. То есть `@/features/works/...` =
`src/features/works/...`.

## 5. Работа с данными (API)

- Базовый адрес сервера: `http://188.126.63.242:1580`
- Фронт и API за одним адресом (nginx), **CORS не нужен**.
- В коде запросы — по относительному пути `/api/...`, домен не указывать:
  ```ts
  fetch("/api/works/")
  ```
- Какие эндпоинты дёргать, что они принимают и возвращают — в твоём ТЗ.
- Бэк сам проверяет права и при отказе возвращает 403 — фронт показывает
  «Нет доступа». Видимость полей определяет бэк: что прислал, то и
  показываем; вручную ничего не прячем.

**Состояния обязательны на каждом экране с данными:**
загрузка / пусто / ошибка. Готовые компоненты — см. п.6.

Пока сервер недоступен — пишем на **моках** в форме будущего ответа API
(те же поля), потом заменим мок на реальный запрос. Моки держи в
`features/<подсистема>/mocks.ts`.

## 6. Что уже есть в проекте (использовать, НЕ выдумывать)

ГОТОВЫЕ ХЕЛПЕРЫ:
- Текущий пользователь: `getCurrentUser()` из `@/lib/mockAuth`.
  Возвращает `{ role, firstName, lastName, middleName, organization }`.
  Поле `role` — строка: "student" / "teacher" / "operator" / "admin".
  ⚠️ Отдельного хука `useAuth` в проекте НЕТ — используй `getCurrentUser`.
- Переключение роли для теста — в шапке справа (dev-инструмент).

ОБЩИЕ КОМПОНЕНТЫ (из `@/components/shared/`) — ТОЧНЫЕ ПРОПСЫ (не выдумывай
другие):
```
PageHeader     { title: string; badge?: string; actions?: ReactNode }
LoadingState   { rows?: number }
EmptyState     { title?: string; description?: string; action?: ReactNode }
ErrorState     { title?: string; description?: string; action?: ReactNode }
InlineSpinner  — без пропсов
```
(У EmptyState/ErrorState поля называются title/description, НЕ message.
 У PageHeader НЕТ поля description.)

UI-КОМПОНЕНТЫ (из `@/components/ui/` — shadcn, не редактировать):
- кнопки, инпуты, таблицы, диалоги, табы, селекты и т.д.

ЭТАЛОННЫЕ ЭКРАНЫ — НЕ ТРОГАТЬ, использовать как ОБРАЗЕЦ:
- `features/contingent/TeacherContingentPage.tsx` — таблица + карточка
- `features/seating/TeacherSeatingPage.tsx` — интерактив + выбор с чипсами
- `features/settings/SettingsPage.tsx` — форма + адаптив
Это готовые рабочие образцы. Смотри на них и делай ПО ИХ ПОДОБИЮ в СВОИХ
новых файлах. НЕ переписывай и не «причёсывай» эталоны — они уже готовы.

МОКИ: создавай свои в `features/<своя-подсистема>/mocks.ts` в форме
ответа API из ТЗ (те же поля). Готовых моков под твою подсистему нет.

## 7. Адаптивность

ОБЯЗАТЕЛЬНО удобны на телефоне (ученики смотрят с телефона):
- Кабинет ученика (список работ, карточка работы), Настройки, лендинг, вход.
  Эти экраны на телефоне должны быть удобными, не просто «не ломаться».

DESKTOP-FIRST (используются за компьютером; на телефоне НЕ должны
ломаться, но идеальный мобильный вид не обязателен):
- Управляющие экраны учителя/оператора/админа: Контингент, Аудитории,
  создание работы, карточка модератора, Банк задач, Обработка ЭМ.

КАК: Tailwind-брейкпоинты — по умолчанию мобильный стиль, через md:/lg:
надстраиваем десктоп (пример: `grid-cols-1 md:grid-cols-3`). Таблицы —
горизонтальный скролл на узком экране. Проверять сужением окна браузера.

## 8. Рабочий процесс (от ТЗ до сдачи)

```
1. Прочитать ТЗ своей подсистемы целиком.
2. Открыть репозиторий, запустить (npm install, npm run dev).
3. Полистать эталонные экраны — понять стиль и паттерны.
4. Создать ветку под задачу:
     git checkout main
     git pull origin main
     git checkout -b feature/название-экрана
5. Взять ПЕРВЫЙ экран из своего ТЗ (он указан в блоке «Первый шаг»).
6. Создать НОВЫЙ файл в своей фиче, подключить маршрут (п.4 выше).
   Эталоны не трогаем — делаем свой новый файл.
7. Написать экран по ТЗ (можно с ИИ). Использовать общие компоненты.
8. Проверить в браузере: данные, переходы, состояния, адаптив.
9. npm run build — убедиться, что собирается.
10. Сдать: git add . / git commit / git push → открыть Pull Request.
11. Лид ревьюит → правки → merge.
12. Взять следующий экран → повторить с п.4.
```

ВАЖНО про API: на первых неделях пишем на МОКАХ. Интеграцию с реальным
API НЕ начинай сам — дождись сигнала лида, что общая авторизация и
API-клиент готовы.

## 9. Тестирование

- **Переключатель ролей** — в шапке справа, по клику на имя. Меняет роль
  без выхода. Dev-инструмент, в проде его не будет.
- **Состояния** — проверяй все три: загрузка, пусто, ошибка.
- **Адаптив** — сужай окно браузера до ширины телефона (см. п.7).
- **Сборка** — `npm run build` перед каждым PR.

## 10. Работа с ИИ-ассистентом

ИИ ускоряет, но не заменяет понимание. Дай ему этот гайд + своё ТЗ.

**ИИ обычно делает хорошо:**
- генерит простые страницы (списки, формы) по ТЗ;
- расставляет состояния и адаптив;
- помогает чинить ошибки сборки.

**Проверяй руками (ИИ часто ошибается):**
- **выдумывает то, чего нет** — хук `useAuth`, несуществующие моки, чужие
  пропсы компонентов (`PageHeader description`, `ErrorState message`).
  Сверяйся с п.6 «Что уже есть».
- **поля API** — может выдумать поле. Сверяй с ТЗ.
- **сложная логика** — создание работы, корректура, конструктор LaTeX:
  делает криво, нужен человек.
- **может начать не с того** — причёсывать эталон вместо своей задачи.
  Начинай со СВОЕГО первого экрана (блок «Первый шаг» в ТЗ).

Правило: **прочитай и пойми то, что выдал ИИ, прежде чем коммитить.**

## 11. Правила репозитория (кратко)

Полностью — в `CONTRIBUTING.md`. Главное:
- **Не пушим напрямую в `main`** — только через ветки и Pull Request.
- Ветки: `feature/...`, `fix/...`, `docs/...`.
- Коммиты с префиксом: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`.
- Один PR = одна задача. Перед PR — `npm run build` проходит.
- Минимум 1 апрув (лид/тех-лид) перед вливанием.
