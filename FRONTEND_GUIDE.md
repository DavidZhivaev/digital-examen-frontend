# FRONTEND_GUIDE

Гайд для фронтенд-команды проекта «Цифровой экзамен».
Этот документ — общий контекст для всех подсистем. Конкретные экраны
описаны в отдельных ТЗ (Фронт 1 — Работы, Фронт 2 — Банк задач + Обработка
ЭМ, Фронт 3 — Контингент + Аудитории). Здесь — как устроен проект и как в
нём работать.

> Если работаешь с ИИ-ассистентом — дай ему этот гайд + своё ТЗ. Тогда он
> будет подсказывать по нашей структуре, а не выдумывать свою.

---

## 1. О проекте

Веб-приложение для проведения семестровых работ в школе: рассадка,
проверка, аналитика. Роли: ученик, учитель, оператор, администратор —
у каждой свой кабинет.

**Стек:**
- React + TypeScript
- Vite — сборка и dev-сервер
- Tailwind CSS + shadcn/ui — стили и компоненты
- React Router — маршрутизация

Понадобятся (ставить по мере необходимости, согласовав с лидом):
- TanStack Query — запросы к API
- react-hook-form + zod — формы и валидация
- react-katex — рендер LaTeX (банк задач)
- react-pdf — просмотр PDF (сканы работ)

## 2. Структура папок

```
src/
  features/                  ← подсистемы; каждый фронт владеет своими
    landing/                 публичная страница
    auth/                    вход
    settings/                настройки аккаунта
    contingent/              Контингент (Фронт 3)
    seating/                 Рассадка
    works/                   Работы (Фронт 1)
    bank/                    Банк задач (Фронт 2)
    processing/              Обработка ЭМ (Фронт 2)
    classrooms/              Аудитории (Фронт 3)
  components/
    ui/                      компоненты shadcn — НЕ редактировать вручную
    layout/                  шапка и общий каркас (AppHeader, AppLayout)
    shared/                  общие компоненты (PageHeader, состояния и т.д.)
  lib/                       утилиты, общий мок авторизации (mockAuth)
  App.tsx                    корневой роутер
```

**Правило:** каждый работает в своей папке `features/<подсистема>/`.
Внутри держи: страницы (`*.tsx`), запросы (`api.ts`), типы (`types.ts`),
моки (`mocks.ts`). Общее (нужное нескольким) — в `components/shared/`.

## 3. Как запустить

```bash
git clone https://github.com/DavidZhivaev/digital-examen-frontend.git
cd digital-examen-frontend
npm install
npm run dev          # откроется http://localhost:5173
```

Проверка, что код собирается (делать перед каждым PR):
```bash
npm run build        # должно быть "✓ built" без ошибок
```

## 4. Как добавить новый экран

На примере экрана «Список работ ученика» (маршрут /student/works):

1. Создай файл в своей фиче:
   `src/features/works/StudentWorksPage.tsx`
   ```tsx
   export function StudentWorksPage() {
     return <div>...</div>
   }
   ```

2. Подключи маршрут в `src/App.tsx`:
   - добавь импорт:
     `import { StudentWorksPage } from "@/features/works/StudentWorksPage"`
   - найди заглушку этого маршрута и замени её:
     ```tsx
     // было:
     <Route path="/student/works" element={<PlaceholderPage title="Мои работы" />} />
     // стало:
     <Route path="/student/works" element={<StudentWorksPage />} />
     ```

3. Открой страницу в браузере (переключив роль, см. п.8).

Алиас `@/` означает папку `src/`. То есть `@/features/works/...` =
`src/features/works/...`.

## 5. Работа с данными (API)

- Базовый адрес сервера: `http://188.126.63.242:1580`
- Фронт и API за одним адресом (nginx), **CORS не нужен**.
- В коде запросы — по относительному пути `/api/...`, домен не указывать:
  ```ts
  fetch("/api/works/")
  ```
- Какие эндпоинты дёргать, что они принимают и возвращают — в твоём ТЗ.
- Бэк сам проверяет права и при отказе возвращает 403 — фронт показывает
  «Нет доступа». Видимость полей определяет бэк: что прислал, то и
  показываем; вручную ничего не прячем.

**Состояния обязательны на каждом экране с данными:**
загрузка / пусто / ошибка. Готовые компоненты — в
`components/shared/states.tsx` (`LoadingState`, `EmptyState`, `ErrorState`).

Пока сервер недоступен — можно писать на **моках** в форме будущего ответа
API (те же поля), потом заменить мок на реальный запрос. Моки держи в
`features/<подсистема>/mocks.ts`.

## 6. Общие компоненты

Не пиши своё, если есть общее. Сейчас доступны:

- `@/components/shared/PageHeader` — заголовок страницы + бейдж + кнопки.
- `@/components/shared/states` — `LoadingState`, `EmptyState`,
  `ErrorState`, `InlineSpinner`.
- `@/components/ui/*` — кнопки, инпуты, таблицы, диалоги и т.д. (shadcn).

Эталонные экраны — смотри как сделано и делай так же:
- Таблица + карточка: `features/contingent/TeacherContingentPage.tsx`
- Интерактив + выбор с чипсами: `features/seating/TeacherSeatingPage.tsx`
- Форма + адаптив: `features/settings/SettingsPage.tsx`

Если видишь, что компонент нужен не только тебе (таблица, поиск, карточка)
— не копируй у соседа, договорись с лидом и вынеси в `components/shared/`.

## 7. Рабочий процесс (от ТЗ до сдачи)

```
1. Прочитать ТЗ своей подсистемы целиком.
2. Открыть репозиторий, запустить (npm install, npm run dev).
3. Полистать эталонные экраны — понять стиль и паттерны.
4. Создать ветку под задачу:
     git checkout main
     git pull origin main
     git checkout -b feature/название-экрана
5. Взять ПЕРВЫЙ экран из ТЗ (от простого к сложному).
6. Создать файл в своей фиче, подключить маршрут (п.4 выше).
7. Написать экран по ТЗ (можно с ИИ). Использовать общие компоненты.
8. Проверить в браузере: данные, переходы, состояния (загрузка/пусто/
   ошибка), адаптив (сузить окно).
9. npm run build — убедиться, что собирается.
10. Сдать:
     git add .
     git commit -m "feat: список работ ученика"
     git push origin feature/название-экрана
    → открыть Pull Request на GitHub.
11. Лид ревьюит → правки → merge.
12. Взять следующий экран → повторить с п.4.
```

## 8. Тестирование

- **Переключатель ролей** — в шапке справа, по клику на имя. Меняет роль
  (ученик/учитель/оператор/админ) без выхода. Это dev-инструмент, в проде
  его не будет. Так быстро проверяешь экран под нужной ролью.
- **Состояния** — проверяй все три: загрузка, пусто, ошибка.
- **Адаптив** — сужай окно браузера до ширины телефона, вёрстка не должна
  ломаться.
- **Сборка** — `npm run build` перед каждым PR.

## 9. Работа с ИИ-ассистентом

ИИ ускоряет, но не заменяет понимание. Дай ему этот гайд + своё ТЗ.

**ИИ обычно делает хорошо:**
- генерит простые страницы (списки, формы) по ТЗ;
- расставляет состояния и адаптив;
- помогает чинить ошибки сборки.

**Проверяй руками (ИИ часто ошибается):**
- **поля API** — ИИ может выдумать поле, которого нет. Сверяй с ТЗ.
- **сложная логика** — создание работы, корректура, конструктор LaTeX:
  ИИ делает криво, нужен человек.
- **общие компоненты** — ИИ склонен копировать чужой стиль вместо
  использования `components/shared`. Следи за этим.
- **права и видимость** — не доверяй догадкам ИИ, бери из ТЗ.

Правило: **прочитай и пойми то, что выдал ИИ, прежде чем коммитить.**
Непонятый код — источник багов, которые потом некому чинить.

## 10. Правила репозитория (кратко)

Полностью — в `CONTRIBUTING.md`. Главное:

- **Не пушим напрямую в `main`** — только через ветки и Pull Request.
- Ветки: `feature/...`, `fix/...`, `docs/...`.
- Коммиты с префиксом: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`.
- Один PR = одна задача. Перед PR — `npm run build` проходит.
- Минимум 1 апрув (лид/тех-лид) перед вливанием.

Вопросы по API и эндпоинтам — к лиду. Не угадывай и не выдумывай поля:
