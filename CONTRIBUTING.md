# Как работать в проекте

## Главное правило

**Не пушим напрямую в `main`.** Вся работа — через ветки и Pull Request.
`main` всегда должен оставаться рабочим (собираться и запускаться).

## Рабочий процесс

```bash
# 1. Обновить main перед началом
git checkout main
git pull origin main

# 2. Создать свою ветку
git checkout -b feature/название-задачи

# 3. Писать код, коммитить
git add .
git commit -m "feat: добавил список работ ученика"

# 4. Запушить свою ветку
git push origin feature/название-задачи

# 5. Открыть Pull Request на GitHub → дождаться ревью → влить в main
```

## Названия веток

- `feature/...` — новая функциональность (`feature/works-student`)
- `fix/...` — исправление бага (`fix/seating-validation`)
- `docs/...` — документация

## Названия коммитов

Используем префиксы:
- `feat:` — новая фича
- `fix:` — исправление
- `refactor:` — переписал без изменения поведения
- `style:` — форматирование, отступы
- `docs:` — документация
- `chore:` — рутина (зависимости и т.п.)

## Pull Request

- Один PR = одна задача
- В описании: что сделал, скриншот если UI
- Минимум 1 апрув (от лида или тех-лида) перед вливанием
- Перед открытием PR убедись, что `npm run build` проходит

## Куда класть код

- Свои экраны — в `src/pages/<роль>/` или `src/pages/`
- Переиспользуемое (таблицы, поиск, состояния) — в `src/components/shared/`, **не копировать у соседа**
- Моки — рядом со своей подсистемой, **в форме будущего ответа API** (чтобы при переходе на бэк менять минимум)

## Общие компоненты

Если видишь, что компонент нужен не только тебе (таблица, карточка, поиск) —
выноси в `shared/`, не пиши свою копию. Если общий компонент уже есть — используй его.
