## 1. Назначение

Дать приложению реальный вход: пользователь логинится → получает токен →
все запросы к API уходят авторизованными. Заменить мок-пользователя
(`mockAuth`) на реального, не сломав код остальных фронтов.

---

## 2. Контракт бэка (готово, не выдумывать)

- **Логин:** `POST /api/auth/login` ← `{ login, password, device_name? }`
  → `{ access_token, refresh_token, token_type: "bearer" }`
- **Передача токена:** заголовок `Authorization: Bearer <access_token>` в
  каждом защищённом запросе.
- **Текущий пользователь:** `GET /api/users/me` → данные юзера (роль, ФИО,
  email и т.д.).
- **Обновление токена:** `POST /api/auth/refresh`
  ← `{ refresh_token, access_token }` → новая пара.
- **Выход:** `POST /api/auth/logout` ← `{ refresh_token }`.

Коды ошибок: `401` — нет/протух токен; `403` — нет прав на действие.

---

## 3. Что сделать (в этом порядке)

1. **Vite-прокси.** В `vite.config.ts` проксировать `/api` →
   `http://188.126.63.242:1580`. Цель: в коде используем относительные пути
   `fetch("/api/...")`, домен не пишем.

2. **API-client** (`src/lib/api.ts`). Обёртка над `fetch`, которая:
   - берёт access-токен из хранилища и кладёт в `Authorization: Bearer`;
   - парсит JSON;
   - на не-2xx — бросает ошибку, в которой **доступен код статуса** (чтобы
     выше отличать `401` / `403` / прочее);
   - экспортирует удобные методы: `api.get`, `api.post`, `api.patch`,
     `api.delete`.

3. **Хранилище токенов** (`src/lib/auth.ts`). Сохранять / читать / чистить
   `access_token` и `refresh_token` в `localStorage` под фиксированными
   ключами.

4. **Логин.** Подключить существующую `LoginPage` к `POST /api/auth/login`:
   отправить логин/пароль → сохранить токены → увести на главную. Показать
   ошибку при неверных данных.

5. **Текущий пользователь.** ⚠️ **Критичный пункт, читай внимательно.**
   Сейчас весь проект использует **синхронный** `getCurrentUser()` из
   `@/lib/mockAuth` (возвращает объект с `role` и ФИО). **Эту сигнатуру
   менять НЕЛЬЗЯ** — на ней держатся экраны Фронтов 2 и 3, асинхронный
   `getCurrentUser` сломает их код.
   Решение:
   - при старте приложения один раз дёрнуть `GET /api/users/me`;
   - положить юзера в стор/контекст;
   - `getCurrentUser()` оставить **синхронным** — пусть читает из стора.

   То есть меняется источник данных внутри, а интерфейс для остальных
   остаётся прежним.

6. **Обработка ошибок.**
   - `401` (токен протух / отсутствует) → редирект на `LoginPage`.
   - `403` (нет прав) → показать «Нет доступа» (компонент `ErrorState` из
     `@/components/shared/states` уже есть).

## 5. Definition of Done

- [ ] Vite-прокси: `fetch("/api/...")` уходит на сервер
- [ ] api-client подставляет `Bearer`-токен в каждый запрос, ошибки несут код
      статуса
- [ ] Логин тестовым аккаунтом работает, токены сохраняются в `localStorage`
- [ ] `getCurrentUser()` отдаёт **реального** юзера с ролью, **сигнатура
      синхронная и не изменилась**
- [ ] `401` → редирект на логин; `403` → «Нет доступа»
- [ ] Файлы — в `src/lib/` и `src/components/` (общий слой), не в `features/`
- [ ] `npm run build` зелёный, PR прошёл ревью

## 6. Что нужно от лида до старта пунктов 4–5

- **Тестовые аккаунты** (логин + пароль) на каждую роль: ученик / учитель /
  оператор / админ — с **уже установленными паролями** (у «новых» аккаунтов
  логин вернёт `403` с требованием задать пароль по ссылке из письма).
- Пункты 1–2 (прокси, api-client) можно делать **до** получения аккаунтов.
